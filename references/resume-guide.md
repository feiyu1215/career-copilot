# 简历优化方法论参考

> **加载上下文**：当 SKILL.md 路由到简历模块时加载本文件。包括：简历诊断、针对性优化、匹配驱动优化、简历格式建议等场景。如果用户只是在匹配岗位或准备面试，不需要加载。

> 本文件是 career-copilot skill 的简历优化方法论参考。核心差异化在于**匹配驱动的简历优化**——利用匹配引擎（smart_score.py）的评分结果，尤其是 risks 字段和低分维度，精准定位简历的薄弱环节，生成针对性的修改建议，并通过验证循环量化优化效果。

---

## 一、简历优化的两种模式

简历优化根据是否有匹配结果，分为两种工作模式：

### 模式 A：独立诊断（无匹配结果时）

适用场景：用户只想优化简历本身，没有跑过 smart_score 匹配流程，或者还没有明确的目标岗位。

工作方式：使用通用诊断框架（第二章），从结构、表达、逻辑、数据支撑等维度全面审视简历质量。

### 模式 B：匹配驱动优化（有 scored_results.json 时）

适用场景：用户已经通过匹配引擎跑出了评分结果，希望针对特定目标岗位优化简历。

工作方式：从 scored_results.json 的 risks 字段精准定位问题，结合 boundary_profile.json 的方向锚点，生成有针对性的修改建议。这是本 skill 区别于通用简历指南的核心能力。

**选择逻辑**：如果工作目录下存在 scored_results.json，优先使用模式 B；否则退回模式 A。两种模式可以组合使用——先用模式 A 做基础体检，再用模式 B 做精准优化。

## 二、通用诊断框架（模式 A 使用）

### 2.1 写简历前的三个思考

在动手修改简历之前，先回答三个问题：

**问题一：目标岗位 JD 说了什么？**

不要凭印象写简历。拿到目标岗位的 JD 后，拆解出三层信息：硬性门槛（学历、年限、必备技能）、核心职责（日常做什么）、加分项（有则更好的能力）。如果用户提供了目标岗位链接，用 web_search 获取完整 JD 文本进行拆解。

**问题二：你的经历和这个岗位匹不匹配？**

把自己的经历逐条对照 JD 的要求，标记出三类：强匹配（直接对口）、可迁移（有相关性但需要桥接叙事）、缺失（确实没有）。这个分类决定了简历中每段经历的详略和呈现角度。

**问题三：你有什么独特的亮点？**

在满足基本匹配之外，你有什么能让面试官记住你的东西？可能是：独特的项目规模、跨领域的组合能力、从 0 到 1 的经历、显著的业务成果等。亮点应该放在简历最显眼的位置。

### 2.2 STAR 法则

STAR 是描述项目经历的基本结构，确保每段经历都有清晰的叙事逻辑：

- **S（Situation）**：背景是什么？面临什么挑战或需求？用一句话交代清楚。
- **T（Task）**：你的具体任务/目标是什么？你负责的范围边界在哪里？
- **A（Action）**：你做了什么？关键决策是什么？技术方案怎么选的？用动词开头，体现主动性。
- **R（Result）**：结果如何？用数据量化。没有精确数据时，用对比（优化前 vs 优化后）或定性描述（上线后获得 XX 好评/被 YY 采纳）。

STAR 不是要求每段经历都写成四段论，而是确保信息完整。简历中通常压缩为 1-2 行，但底层逻辑要完整。

**示例**：
- 弱：负责推荐系统优化
- 强：针对首页推荐 CTR 下降问题（S），主导召回策略重构（T），引入多路召回融合 + 实时特征工程（A），CTR 提升 12%，日均 GMV 增长 800w（R）

### 2.3 常见简历误区

| 误区 | 表现 | 改进方向 |
|------|------|----------|
| 信息罗列没有重点 | 所有经历平均用力，看不出核心竞争力 | 按目标岗位相关度排序，核心经历详写，其余一笔带过 |
| 大量堆砌缺少逻辑 | 技能列了 20 个，项目写了 8 段，没有主线 | 围绕 2-3 个核心能力主线组织，每个主线下挂 1-2 个代表项目 |
| 描述空洞 | "参与了 XX 项目""负责 XX 模块" | 写清楚做了什么决策、解决了什么问题、产出了什么结果 |
| 缺少数据支撑 | "显著提升了性能""大幅降低了成本" | 换成具体数字：提升 30%、降低 500ms、节省 200w/年 |
| 一份简历走天下 | 投不同方向的岗位用同一份简历 | 根据目标岗位调整经历的呈现角度和详略 |
| 逻辑不清晰 | 时间线混乱、因果关系不明 | 按时间倒序，每段经历内部保持"问题→方案→结果"逻辑 |

### 2.4 常见表达优化对照

| 原始表达 | 问题 | 优化后 |
|----------|------|--------|
| 负责后端开发 | 太笼统，看不出做了什么 | 负责 XX 服务的架构设计与核心模块开发，支撑日均 XXw 请求 |
| 参与了 XX 项目 | "参与"是最弱的动词 | 主导/设计/推动/落地 XX 项目的 YY 部分 |
| 优化了系统性能 | 没有量化 | 通过 XX 手段将 P99 延迟从 200ms 降至 50ms |
| 使用了 XX 技术 | 只说用了什么，没说解决什么问题 | 引入 XX 技术解决 YY 问题，带来 ZZ 收益 |
| 熟悉 XX | 没有证据 | 基于 XX 构建了 YY 系统，线上稳定运行 Z 个月 |
| 团队协作能力强 | 空洞的自我评价 | 协调 X 个团队 Y 人完成 Z 项目，按期交付 |
| 有丰富的经验 | 没有支撑 | X 年 YY 领域经验，主导过 Z 个从 0 到 1 项目 |

### 2.5 格式建议

**基本原则**：

- 控制在 1-2 页，应届生 1 页，3 年以上可以 2 页
- PDF 格式投递，避免 Word 排版错乱
- 字体统一，正文 10-11pt，标题 12-14pt
- 留白适度，不要塞得满满当当
- 联系方式完整：手机、邮箱、GitHub/博客（如有）

**模块顺序建议**：

- 应届/少经验：教育背景 → 实习/项目经历 → 技能 → 其他
- 有工作经验：个人信息 → 工作经历 → 项目经历（可选）→ 教育背景 → 技能

**注意事项**：

- 避免使用表格嵌套（很多 ATS 系统解析会出错）
- 避免使用图片/图标代替文字
- 技能列表按相关度排序，不要按字母表排
- 每段工作经历控制在 3-5 个 bullet point

## 三、匹配驱动优化框架（模式 B 使用）

这是本方法论的核心差异化章节。当用户已有 scored_results.json 时，我们可以做到"对症下药"式的简历优化。

### 3.1 核心流程

```
scored_results.json 中的目标岗位
    │
    ├── match_reasons → 简历中的优势点（保持/强化）
    └── risks → 简历中的薄弱点
         │
         ├── 分类判断：表述问题 还是 能力缺失？
         │    ├── 表述问题（有经验但没写好）→ 措辞优化建议
         │    └── 能力缺失（确实没有相关经验）→ 迁移叙事策略
         │
         └── 生成针对性修改建议
              │
              └── 验证：重跑匹配，对比分数变化
```

### 3.2 诊断决策树

拿到一个 risk 之后，第一步是判断它的性质：

**表述问题的信号**：

- 用户口头沟通时能说出相关经验，但简历上没体现或一笔带过
- 简历中有相关项目，但没有突出该维度的贡献
- 关键词缺失：JD 要求的技术/能力词在简历中完全没出现
- 角度偏差：经历本身是匹配的，但描述角度偏了（比如 JD 要"系统设计能力"，简历只写了"编码实现"）

**能力缺失的信号**：

- 用户确认自己没有该领域的直接经验
- 简历中找不到任何可以关联的项目或技能
- 该 risk 涉及的是硬性门槛（如特定行业经验、特定证书）

**判断方法**：如果无法确定，向用户提问——"关于 [risk 描述]，你之前有没有做过相关的事情，哪怕不是直接对口的？"用户的回答决定走哪条优化路径。

### 3.3 表述问题的优化策略

当诊断为表述问题时，有三种优化手段：

**策略一：STAR 重写**

找到简历中对应的经历段落，用 STAR 法则重新组织，确保与 risk 相关的贡献被清晰呈现。

示例：
- Risk："缺乏大规模系统设计经验"
- 简历原文："负责订单服务开发"
- 重写后："设计并落地订单服务的分库分表方案，支撑日均 500w 单量，P99 延迟控制在 80ms 以内"

**策略二：量化补充**

很多时候用户有成果但没有量化。引导用户回忆数据：处理了多少数据量？服务了多少用户？提升了多少百分比？节省了多少成本/时间？

**策略三：关键词对齐**

从 JD 中提取高频关键词，检查简历中是否自然地包含了这些词。注意是"自然地"——不是机械堆砌，而是在描述真实经历时使用 JD 中的术语。

### 3.4 能力缺失的应对策略

当诊断为能力缺失时，不能凭空编造，但可以通过以下策略降低 risk 的影响：

**策略一：迁移叙事**

找到用户经历中与缺失能力"相似"的经验，通过类比建立桥接。关键是找到底层能力的共通性。

示例：
- Risk："缺乏机器学习工程经验"
- 用户实际经历：做过大规模数据管道 + 读过 ML 论文 + 有 AB 实验经验
- 迁移叙事："构建了日处理 TB 级数据的实时管道，为下游模型训练提供特征数据；主导 AB 实验平台搭建，具备模型效果评估的完整认知"

**策略二：学习能力展示**

如果缺失的能力是可学习的（非数十年积累型），在简历中展示快速学习的证据：短时间内掌握新技术并产出成果的案例。

**策略三：项目补充建议**

如果是核心能力缺失且无法迁移，坦诚告知用户，并建议通过以下方式补充：
- 开源项目贡献
- Side project
- 在线课程 + 实践项目
- 内部转岗/轮岗获取经验

这属于中长期建议，短期内简历本身改不了这个 risk。

### 3.5 优势点的强化

不要只关注 risks。scored_results 中的 match_reasons 告诉我们哪些是用户的优势。对这些点：

- 确认它们在简历中的位置是否足够醒目（放在前面、用更多篇幅）
- 确认描述是否充分展开（不要因为觉得"这个我很强所以不用多写"而一笔带过）
- 确认量化数据是否到位

### 3.6 与 boundary_profile.json 的联动

boundary_profile.json 提供了用户的能力画像，与简历优化有直接关系：

**direction_anchors（方向锚点）**：

告诉你用户的核心职业方向是什么。简历应该围绕这个方向组织主线叙事。如果 direction_anchors 是"后端架构"，那么简历的核心故事线应该是"我是一个有架构能力的后端工程师"，所有经历都围绕这个定位来呈现。

**signal_words（信号词）**：

告诉你哪些关键词对用户的方向很重要。这些词应该自然地出现在简历中——不是硬塞，而是在描述真实经历时使用这些术语。如果 signal_words 包含"高可用""分布式""容量规划"，检查简历中是否有这些词的合理出现。

**hard_negatives + adjacent_but_different（能力边界）**：

告诉你用户的能力边界在哪里。`hard_negatives` 是完全不相关的方向（如"纯前端开发""市场营销"），`adjacent_but_different` 是看起来相关但角色本质不同的方向（如"算法工程师 vs AI产品经理"）。对于边界之外的领域，如果 JD 要求了，那就是真正的能力缺失，不要试图硬凑。

## 四、验证循环

简历优化不应该是一次性动作。通过匹配引擎，我们可以量化验证优化效果。

### 4.1 验证流程

```
步骤 1：用户根据建议修改简历
步骤 2：重新运行 gen_profile.py 生成新的 boundary_profile.json
步骤 3：重新运行 smart_score.py 对同一批 JD 评分
步骤 4：对比前后 scored_results
步骤 5：判断优化效果
```

### 4.2 对比维度

对比前后两次 scored_results 时，关注以下维度：

- **总分变化**：目标岗位的 overall_score 是否提升？
- **risks 数量变化**：之前标记的 risks 是否减少了？
- **risks 内容变化**：同一个 risk 的描述是否变弱了（从"完全缺乏"变为"经验有限"）？
- **match_reasons 变化**：是否新增了匹配理由？
- **副作用检查**：有没有引入新的 risks？（比如为了强调 A 能力，把 B 能力的描述压缩了）

### 4.3 效果判断

- **明显改善**（目标岗位分数提升 5+ 分，risks 减少 1-2 个）→ 优化有效，确认方向正确
- **轻微改善**（分数提升 1-3 分，risks 表述变弱但没消除）→ 方向对但力度不够，继续迭代
- **无改善**（分数不变，risks 不变）→ 可能是能力缺失问题而非表述问题，调整策略
- **反向变化**（分数下降或新增 risks）→ 修改有副作用，需要回退并重新审视

### 4.4 迭代节奏建议

- 每次修改后建议只改 1-2 个 risks 对应的内容，避免大面积改动后无法归因
- 建议保留每次修改的版本（简历 v1、v2、v3...）和对应的 scored_results
- 一般 2-3 轮迭代即可达到较好效果

## 五、针对性优化的实操模板

### 5.1 单个 Risk 的诊断与建议模板

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
风险点：[直接引用 scored_results 中的 risks 字段文本]
来源岗位：[对应的 JD 标题/公司]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

诊断结果：[表述问题 / 能力缺失 / 待确认]

如果是表述问题：
  当前简历中的相关段落：[引用简历原文]
  问题所在：[为什么匹配引擎认为这是 risk]
  修改建议：[用 STAR 重写的版本，或补充量化数据，或关键词对齐]
  修改后预期：该 risk 应被消除或显著弱化

如果是能力缺失：
  差距分析：[用户有什么，JD 要什么，gap 在哪]
  短期策略：[迁移叙事方案，如有可迁移经验]
  中期策略：[项目补充建议，如有必要]
  坦诚评估：[这个 risk 是否是 deal-breaker？是否建议更换目标岗位？]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 5.2 整体优化报告模板

```
简历优化报告
目标岗位：[岗位名称 @ 公司]
当前匹配分：[overall_score]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

优势点（保持/强化）：
  1. [match_reason_1] → 建议：[保持/强化措施]
  2. [match_reason_2] → 建议：[保持/强化措施]

薄弱点（需要优化）：
  1. [risk_1] → 诊断：[表述/缺失] → 优先级：[高/中/低]
  2. [risk_2] → 诊断：[表述/缺失] → 优先级：[高/中/低]

修改优先级排序：
  先改 [risk_X]（表述问题，改动成本低，预期收益高）
  再改 [risk_Y]（表述问题，需要用户提供更多信息）
  暂缓 [risk_Z]（能力缺失，短期无法通过简历修改解决）

下一步行动：
  [具体的第一步修改建议]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 5.3 优先级排序逻辑

当有多个 risks 需要优化时，按以下优先级排序：

1. **高优先级**：表述问题 + 该维度在 JD 中被反复强调 → 改动成本低、预期收益高
2. **中优先级**：表述问题 + 该维度是加分项而非核心要求 → 值得改但不是最急的
3. **低优先级**：能力缺失 + 有迁移叙事空间 → 需要更多思考和用户信息
4. **暂缓处理**：能力缺失 + 无迁移可能 → 考虑是否该调整目标岗位

## 六、正直原则（红线）

简历优化的底线是真实。以下行为绝对禁止：

**禁止编造项目经历**：没做过的项目不能写。哪怕是"行业内常见的项目"也不能假装自己做过。面试时一旦深挖细节就会暴露。

**禁止夸大数据**：提升 5% 不能写成 50%。服务 10w 用户不能写成 100w。数据造假是最容易被验证的谎言。

**禁止虚构职位/职责**：不能把"参与"写成"主导"，除非你确实是主要决策者。不能给自己加 title。

**如实优化 ≠ 造假**：优化是在真实经历的基础上，选择更好的呈现角度和表达方式。这和造假有本质区别。

**面试深挖的风险**：造假的简历在面试中会被快速识破。面试官会追问细节——"你说提升了 30%，具体是怎么衡量的？""这个架构决策是谁做的？为什么这么选？"。如果答不上来，不仅这个岗位没戏，还可能进入公司黑名单。

**正确的做法**：

- 有的写好，没有的不写
- 对模糊的数据用范围而非精确值（"约 20-30% 的提升"）
- 对自己角色不清的项目，如实描述参与程度（"协助设计""在 XX 指导下完成"）
- 如果简历实在撑不起目标岗位，那说明需要的是积累经验，而不是美化简历

## 七、记忆联动

### 7.1 为什么需要记忆

简历优化往往是一个跨多次会话的过程。用户不会一次性改完所有内容。记忆系统帮助我们：
- 避免重复诊断同一个问题
- 追踪优化历史，看到改进趋势
- 记住用户提供过的背景信息（口头描述的经历等）

### 7.2 记录时机与内容

**记录时机**：每次简历修改完成后。

**记录内容建议**（写入 career-memory，type: resume_update）：

```
{
  "type": "resume_update",
  "date": "YYYY-MM-DD",
  "trigger": "匹配 XX 岗位时发现 risk: [具体 risk]",
  "changes": [
    {
      "section": "工作经历 - XX 公司",
      "before_summary": "原来只写了'负责 XX 模块开发'",
      "after_summary": "重写为 STAR 结构，补充了量化数据",
      "target_risk": "缺乏系统设计经验"
    }
  ],
  "effect": "重跑匹配后分数从 72 提升到 78，该 risk 消除"
}
```

### 7.3 历史记录的使用

下次优化简历时，先查阅历史记录：
- 避免改回之前已经优化过的表述
- 如果同一个 risk 反复出现，说明之前的修改力度不够，需要更深层的调整
- 如果用户之前口头提供过的经历信息，可以直接引用，不需要再次询问

### 7.4 与 boundary_profile 的关系

boundary_profile 是用户能力画像的"快照"。每次简历大改后重新生成的 boundary_profile 可能会变化——这不是 bug，而是因为画像是从简历文本中提取的。记录每次 boundary_profile 的变化，可以帮助用户理解"匹配引擎看到的我"是如何演变的。
